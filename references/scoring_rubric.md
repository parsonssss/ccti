# CCTI 评分细则

读完一段会话历史后,Claude 按以下规则给用户打分。

---

## Step 1 — 4 轴判定(决定 archetype 字母组合)

每一轴二选一,看 user 消息(不是 Claude 的)的整体特征。

### 轴 1:F (Field Commander) vs I (Idea Person)

| 信号 | 倾向 F | 倾向 I |
|---|---|---|
| user 是否在消息里贴具体代码 / diff 片段 | 是 | 否 |
| user 是否描述了"实现方法"层面的细节(用什么函数 / 数据结构) | 是 | 否 |
| user 是否说过"我自己写"「我来 commit」之类 | 是 | 否 |
| user 的需求颗粒度 | 改这行 / 这个函数 | 这个功能 / 这种体验 |
| user 是否要求 Claude 给"实现细节交付" | 否 | 是 |

**判定**:看哪边的信号 ≥ 60%。

### 轴 2:T (Terse) vs V (Verbose)

| 信号 | 倾向 T | 倾向 V |
|---|---|---|
| 单条 user 消息中位字数 | < 30 字 | > 80 字 |
| 是否经常用「下一个」「按这个改」「不要」「下一个」这类 1–4 字命令 | 是 | 否 |
| 是否在 prompt 里给大量 context / 背景 / 引用 | 否 | 是 |
| 标点符号、敬语、礼貌词频率 | 极低 | 正常 |

### 轴 3:S (Sprint) vs M (Marathon)

| 信号 | 倾向 S | 倾向 M |
|---|---|---|
| user 收到回复后多快发下一条(估算消息间隔) | 秒级 | 分钟级以上 |
| 单 session 内推动的不同 task / 功能数 | ≥ 6 | ≤ 3 |
| 是否频繁说"算了""换一个""下一个""快" | 是 | 否 |
| 是否在一个 task 上反复打磨 | 否 | 是 |

### 轴 4:P (Paranoid) vs R (Reckless)

| 信号 | 倾向 P | 倾向 R |
|---|---|---|
| user 是否经常验证 Claude 的输出再继续 | 是 | 否 |
| 是否要求看 diff / 测试 / 截图后才放下条 | 是 | 否 |
| 是否经常在 Claude 完成后直接接下一个需求 | 否 | 是 |
| 是否说过"不验证""yolo""试试再说" | 否 | 是 |

---

## Step 2 — 8 项学术指标评分(0–10 分)

每项给一个分,带一位小数。映射到 archetypes.json 的 dimensions 字段。

### 任务表达清晰度(task_clarity)

- 9–10:几乎每条 prompt 都有 verb + scope,Claude 不需要反问
- 7–8:大多清晰,偶尔需要 Claude 反问澄清
- 5–6:经常含糊("看着办""你懂的")
- < 5:难以从 prompt 提取动作

### 产品思维(product_thinking)

- 9–10:主动谈定价 / 用户体验 / 商业化 / 上架策略
- 7–8:从用户/产品视角描述需求("用户应该能看到 X")
- 5–6:主要谈技术实现
- < 5:无产品维度

### 问题诊断能力(diagnostics)

- 9–10:bug 描述精准到能直接定位 root cause
- 7–8:能给具体症状 + 复现路径
- 5–6:描述模糊("不工作""坏了")
- < 5:无能力定位

### 技术决策(technical_decisions)

- 9–10:在多个选项里果断拍板,有理由
- 7–8:主动放弃低优先项,聚焦核心
- 5–6:跟随 Claude 建议,不主动选
- < 5:反复摇摆

### 迭代节奏(iteration_cadence)

- 9–10:平均 1–2 个 turn 一个新需求,推进流畅
- 7–8:有规律,但偶尔 batch 过大
- 5–6:节奏不稳
- < 5:卡在一处不前进

### 代理协作意识(agent_collaboration)

- 9–10:出问题时打回 Claude,清晰指出,不绕开
- 7–8:多数时候打回,偶尔自己改
- 5–6:经常不告诉 Claude 哪里错,自己默默改
- < 5:不视为协作

### 上下文提供(context_provision)

- 9–10:路径 / 资源 / 截图 / 背景全面
- 7–8:必要信息齐
- 5–6:有时缺关键信息,要 Claude 反问
- < 5:几乎不给 context

### 工程规范意识(engineering_discipline)

- 9–10:主动 tag / changelog / 文档 / 测试
- 7–8:有规范习惯但偶尔跳过
- 5–6:不主动维护
- < 5:无规范习惯

---

## Step 3 — 综合分 + Tier

把 8 项加权平均(每项权重 1/8),映射到 Tier:

| 分数区间 | Tier |
|---|---|
| ≥ 8.5 | S+ |
| 8.0 – 8.5 | S |
| 7.0 – 8.0 | A |
| 6.0 – 7.0 | B |
| < 6.0 | C |

---

## Step 4 — 8 项刻画级生成

这些数据**不要假装精确测量**。基于 archetype 字母组合和 archetypes.json 里的 `signature_stat_templates`,选与之匹配的值。允许 Claude 在 ±20% 内做小幅调整以贴合该用户实际行为。

刻画级核心目的是**让用户对号入座**,而不是真给统计数字。哪怕真实是 12 字它写 13 字,也没人较真。

---

## Step 5 — 选 archetype 卡片内容

根据 4 轴字母组合查 archetypes.json → 拿到 name_cn / name_en / tier(可被 Step 3 算出的实际 tier 覆盖) / slogan / explanation / accent_lines。

填入 template.html,渲染。

---

## 注意

- **没有"正确"或"错误"型** — 16 型都是中性描述,不是评级。Tier 评级独立于 archetype 类型,两个不同 archetype 的用户都可能拿到 S+ 或 C。
- **保持荒诞但克制** — SBTI 风格的核心是"准确得让你尴尬",不是"嘲讽"。刻画级数据是为了让用户笑着说「卧槽这就是我」,不是「这写的什么垃圾」。
